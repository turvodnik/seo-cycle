# E-E-A-T Audit

Cross-cutting checklist для **Experience, Expertise, Authoritativeness, Trust**. Применяется в Phase 1 (audit baseline), Phase 4 (Entity Map — author/citation slots), Phase 6 (writing QA).

Активируется через `eeat.enabled: true` в `seo-cycle.yaml`.

## Когда критично

- **Обязательно**: Your Money Your Life (YMYL) — медицина, финансы, юриспруденция, страхование, безопасность
- **Очень важно**: образование, родительство, новости, технические обзоры
- **Важно**: ecommerce с экспертной составляющей (строительство, автомобили, электроника)
- **Полезно**: блоги, медиа, B2B SaaS
- **Менее критично**: развлечения, fashion (но trust = всегда)

## Опросник перед написанием/публикацией

### Experience (опыт)

Реальный опыт автора с темой — практическое использование, не теоретическое знание.

- [ ] **Автор использовал/делал это лично?** Если да — описать в bio + first-person примеры в статье
- [ ] **Есть фото процесса / результата?** (не stock)
- [ ] **Указано «когда»** — год опыта, дата теста, версия продукта
- [ ] **Если групповой опыт** (команда, бригада) — кто конкретно делал
- [ ] **First-hand признаки в тексте**: «я заметил», «когда мы делали», «после 3 лет использования»

**Запреты:**
- Не выдавать чужой опыт за свой
- Не использовать ChatGPT-style abstract («это надёжный материал, который зарекомендовал себя»)
- Не писать про опыт которого нет — лучше пригласить эксперта или сослаться

### Expertise (экспертиза)

Формальные / профессиональные квалификации.

- [ ] **Автор имеет образование/сертификат по теме?** Указать в bio
- [ ] **Кол-во лет работы в области?** (конкретное число)
- [ ] **Публикации / выступления / награды?** Линки
- [ ] **Команда — кто специалисты?** (для медицинских/финансовых статей — обязательно подпись с регалиями)
- [ ] **Источники — на профильные публикации, не на блоги**: ГОСТы, peer-reviewed журналы, отраслевые отчёты

**Запреты:**
- Фейковые credentials («ведущий эксперт» без подтверждения)
- Громкие звания без проверки («сертифицированный специалист» без названия сертификата)

### Authoritativeness (авторитет)

Признание в индустрии (внешние сигналы).

- [ ] **Внешние ссылки на сайт от авторитетных источников** (см. backlink audit)
- [ ] **Упоминания бренда без ссылки** (brand mentions) в авторитетных публикациях
- [ ] **Сертификации / партнёрства** — членство в ассоциациях, official partner с производителями
- [ ] **Социальные доказательства** — реальные отзывы (Google, Я.Бизнес, Trustpilot — не on-site)
- [ ] **Case studies** — конкретные проекты с цифрами
- [ ] **Citations в Wikipedia / профильных wiki** — если уместно

**Запреты:**
- Фейковые отзывы или AggregateRating
- «Партнёрство» без официального документа
- Звания «лидер рынка» без отраслевой статистики

### Trust (доверие)

Базовое — фундамент для всех остальных трёх.

- [ ] **HTTPS** (без mixed content)
- [ ] **About page** с реальной информацией: физический адрес, ОГРН/INN (для РФ), команда с фото
- [ ] **Contacts** — телефон + email + физический адрес. Не только форма
- [ ] **Privacy policy / Terms** — заполненные, не lorem
- [ ] **Cookies banner** — корректный консент
- [ ] **Domain age + WHOIS** не приватный (для коммерческих сайтов)
- [ ] **Schema.org Organization** — `address`, `telephone`, `sameAs` (соцсети)
- [ ] **Footer** — copyright год, юр. инфо, ссылки на политики
- [ ] **404 page** — кастомная с навигацией, не пустая
- [ ] **Reviews / testimonials с реальными именами**
- [ ] **Цены прозрачные** (если ecommerce/услуги)
- [ ] **Возврат / гарантия** — указаны явно
- [ ] **Платёжные методы** — иконки реальных (Visa, MIR), без фейков

## Phase 1 (audit baseline)

Прогон по всему сайту:

```bash
# Чек-лист в виде вопросов — заполнять руками на основе обхода:
# 1. About page есть? Реальные авторы?
# 2. Statys footer (copyright, контакты)?
# 3. Privacy policy / Terms — присутствуют, не lorem?
# 4. HTTPS, нет mixed content?
# 5. Schema.org Organization корректна?
# 6. Reviews с именами или фейковые stars?
# 7. Author bios на статьях (если блог)?
# 8. Sources / citations в технических статьях?
```

Выход: `01-audit.md` секция «E-E-A-T baseline» со списком найденных проблем и приоритетами.

## Phase 4 (Entity Map slots)

В шаблоне Entity Map (`templates/entity-map.template.md`) — добавь frontmatter поля:

```yaml
eeat:
  author:
    name: "Имя Фамилия"
    bio_url: /authors/<slug>/
    credentials: ["диплом строит. вуза", "10 лет на стройке", "сертификат Knauf"]
  experience: "first-hand"      # first-hand | theoretical | aggregated
  citations:
    - "ГОСТ 9573-2012"
    - "СП 50.13330.2012"
  last_reviewed_by: "<имя эксперта или редактора>"
  review_date: 2026-05-26
```

## Phase 6 (writing QA)

Перед публикацией — пробежать чек-лист этого документа. Особое внимание к YMYL темам:
- Author bio с подтверждаемыми credentials
- Минимум 3 источника на каждое числовое утверждение
- Дата последней проверки на странице (visible to user)
- Reviewer (для медицинских — врач, для финансовых — финансист)

## Phase 9 monitoring сигналы E-E-A-T

- **Resp. Brand mentions** — рост (через Google Alerts, Mention.com, ahrefs)
- **Linking root domains** — рост (не флэт)
- **Direct traffic** — рост (signal trust)
- **Bounce rate** на YMYL-страницах — падает с улучшением E-E-A-T

## Anti-patterns (что НЕ делать)

| Anti-pattern | Почему плохо |
|---|---|
| AI-generated bio автора без реального человека | Google детектит, доверие падает |
| Stock-фото «улыбающаяся команда» | Distrust signal |
| «Эксперт с 20+ годами опыта» без имени | Anonymous expertise = no expertise |
| Reviews / testimonials с stock-аватарками | Fake signals |
| Цитаты Wikipedia как основной источник для технических утверждений | Wikipedia ≠ primary source |
| Sources секция в конце с битыми/устаревшими ссылками | Trust gap |
| Бесконечные «awards» без подтверждения | Authority inflation |
